Odara Documentation

The AI-first, open-source ETL/ELT platform built in Rust and React. Design data pipelines visually, transform with SQL and Python, and orchestrate at scale.

Rust Backend React Frontend Apache Arrow DataFusion SQL 6+ Databases Cloud Storage AI-Powered Open Source

1. Getting Started

1.1 What is Odara?

Odara is an AI-first, open-source ETL/ELT platform designed to disrupt traditional enterprise data integration tools such as Talend, Informatica, and SSIS. It combines a high-performance Rust backend with a modern React frontend to deliver a visual pipeline editor that is fast, extensible, and developer-friendly.

At its core, Odara uses Apache DataFusion as its SQL engine, which operates on Apache Arrow columnar memory format. This means data transformations happen in-memory at native speed without serialization overhead. For more flexible transformations, Odara also supports Python via subprocess execution with PyArrow for zero-copy data exchange.

Key characteristics that distinguish Odara from other ETL platforms:

  • Performance-first architecture — Written in Rust with no garbage collection pauses. Execution is asynchronous and concurrent using Tokio.
  • Visual pipeline editor — Built with React Flow, enabling drag-and-drop pipeline design with real-time execution feedback via Server-Sent Events (SSE).
  • Polyglot transforms — Mix SQL and Python in the same pipeline. SQL executes in DataFusion; Python runs in a subprocess with pandas.
  • Small blocks philosophy — Each node does one thing well. Complex pipelines are assembled by composing small, focused components.
  • AI-assisted development — Integrated AI chat with support for DeepSeek, OpenAI, Anthropic, and Google Gemini to generate SQL, Python, and entire pipelines from natural language.
  • Lightweight persistence — SQLite with WAL mode for metadata, pipeline definitions, execution history, and context variables. No heavy database infrastructure required.
  • Encrypted credentials — All connection credentials are encrypted at rest using AES-GCM.
Community Edition This documentation covers the Odara Community Edition, which is free and open-source under the Apache 2.0 license.

1.2 Architecture

Odara follows a modular crate-based architecture. Each crate has a clear responsibility and communicates through well-defined interfaces.

CrateResponsibility
odarax-corePipeline execution engine, DataFusion integration, file connectors (CSV, Excel, XML, Magic File), cloud connectors (S3, Google Drive, Azure Blob), FTP/SFTP, REST client, Maestro orchestrator, Python subprocess executor, context management
odarax-apiHTTP API server built on Axum. Routes, handlers, SSE streaming, SQLite storage layer, file browsing, schema inference
odarax-connectorsDatabase connectors: PostgreSQL, MySQL, Oracle, SQL Server, MongoDB, Snowflake. Source and target operations with bulk loading
odarax-aiAI integration layer. DeepSeek, OpenAI, Anthropic, and Gemini API clients. Prompt templates for SQL, Python, and pipeline generation
odarax-schedulerJob scheduling with cron expressions. Triggers pipeline and maestro executions on schedule

The frontend is a single-page React application that communicates with the backend exclusively through REST API calls and SSE connections. All data between pipeline nodes passes as Apache Arrow RecordBatch objects in memory, ensuring zero-copy transfers for SQL operations and efficient serialization for Python subprocess communication.

The execution flow follows this path:

  1. User designs pipeline visually in the React Flow canvas
  2. Pipeline definition is saved to SQLite via the REST API
  3. User triggers execution; the API creates a CancellationToken and spawns the executor
  4. The executor traverses the DAG topologically, executing each node
  5. Progress events stream back to the frontend via SSE
  6. Results (row counts, errors, previews) are stored in SQLite for later review

1.3 Installation

Odara is distributed as a pre-built binary — no compilation, no Rust, no Node.js required. Just download, install, and run.

Ubuntu / Debian (amd64)

# 1. Download the .deb package
wget https://github.com/odara-data/odara-community/releases/latest/download/odara_0.1.0_amd64.deb

# 2. Install
sudo dpkg -i odara_0.1.0_amd64.deb

# If you get dependency errors:
sudo apt-get install -f

# 3. Start Odara
odara

Odara starts on port 3002 and automatically opens your browser at http://localhost:3002.

Windows (amd64)

  1. Download odara_0.1.0_windows_amd64.zip from the Releases page.
  2. Right-click the .zip file and select Extract All... to a folder (e.g., C:\Program Files\Odara).
  3. Double-click start.bat to start.

Odara starts on port 3002 and automatically opens your browser at http://localhost:3002.

Tip If Windows Firewall prompts you, click Allow access to let Odara listen on the network.

macOS (Apple Silicon / ARM64)

# 1. Download
curl -LO https://github.com/odara-data/odara-community/releases/latest/download/odara_0.1.0_macos_arm64.tar.gz

# 2. Extract
tar -xzf odara_0.1.0_macos_arm64.tar.gz

# 3. Remove macOS quarantine flag (required for unsigned binaries)
xattr -rd com.apple.quarantine odara-macos-arm64/

# 4. Start Odara
./odara-macos-arm64/odara

# Optional: copy to your PATH for global access
sudo cp odara-macos-arm64/odara /usr/local/bin/
odara

Python Setup (Optional)

Python is only required if you use Python Transform nodes. If you only use SQL transforms, file connectors, and database connectors, Python is not needed.

  1. Install Python 3.9+ from python.org
  2. On Windows: make sure to check "Add Python to PATH" during installation
  3. Install the PyArrow library: 
    pip install pyarrow
  4. Verify: 
    python --version    # Should show Python 3.9+
python -c "import pyarrow; print(pyarrow.__version__)"
Note On Windows, if python opens the Microsoft Store instead of running Python, go to Settings > Apps > Advanced app settings > App execution aliases and toggle off both python.exe and python3.exe.

1.4 Quick Start

Follow this 5-minute tutorial to create and run your first pipeline.

  1. Start Odara. Run ./start.sh (Linux/macOS) or double-click start.bat (Windows). Your browser will open automatically at http://localhost:3002.
  2. Create a project. Click the project dropdown at the top-left of the editor, select "New Project", enter a name (e.g., "My First Project"), and choose a workspace directory.
  3. Open the Nodes palette. Click the "Nodes" tab in the left sidebar. You will see three categories: INGEST, TRANSFORM, and DELIVER.
  4. Add a CSV Source. Drag "CSV Source" from INGEST onto the canvas. Click the node to open the Properties panel. Set the file path to a CSV file on your system.
  5. Add a SQL Transform. Drag "SQL Transform" from TRANSFORM onto the canvas. In the code editor, write a query such as: 
    SELECT * FROM input WHERE amount > 100 ORDER BY amount DESC
  6. Add a Console Target. Drag "Console" from DELIVER onto the canvas.
  7. Connect the nodes. Drag from the output port (right side) of CSV Source to the input port (left side) of SQL Transform. Then connect SQL Transform to Console.
  8. Run the pipeline. Click the "Run" button in the toolbar. Watch nodes turn green as they complete. Open the Console panel at the bottom to see the output data.
Note The input table name in SQL Transform refers to the data arriving from the upstream node. If a node has multiple inputs, each connection is named after the source node.

2. Projects

2.1 Creating a Project

Every workspace in Odara begins with a project. A project is a self-contained unit that holds all your pipelines, maestros, resources, variables, and execution history.

Project selector dropdown with list of projects and settings access

To create a new project:

  1. Click the project dropdown at the top-left corner of the interface.
  2. Select "New Project" from the menu.
  3. Enter a project name and select or create a workspace directory on your filesystem.
  4. Click "Create". Odara initializes the project structure with a SQLite database and default configuration.

2.2 Project Selector

The project selector dropdown at the top-left of the editor shows all available projects. Click any project to switch to it immediately. All pipelines, resources, and state associated with that project load into the editor.

You can also open a project by its filesystem path using the "Open by Path" option. This is useful for projects created outside of Odara or shared via Git.

2.3 Project Settings

Access project settings through the gear icon in the project dropdown. Settings include:

  • AI Provider — Choose between DeepSeek, OpenAI, Anthropic, or Google Gemini for AI-assisted features.
  • AI Model — Select the specific model to use (e.g., gpt-4, claude-3, deepseek-chat).
  • API Key — Enter your AI provider API key (stored securely).
  • Default Environment — Set which environment (Dev, QA, Prod) is active by default.

2.4 Project Structure

Each project creates the following structure on disk:

{project_path}/
  data/
    odara.db            # SQLite database (WAL mode)
    odara.db-wal        # Write-ahead log
    odara.db-shm        # Shared memory file
    screenshots/        # Auto-captured pipeline screenshots
      pipeline_{id}.png
      maestro_{id}.png

The SQLite database contains all pipeline definitions, node configurations, metadata connections, context variables, execution history, and scheduling rules. Pipeline screenshots are automatically captured each time you save, providing a visual record for documentation.

3. Editor

3.1 Interface Overview

The Odara editor is divided into several panels that work together to provide a comprehensive pipeline design environment.

Full editor interface: canvas with a multi-output pipeline, sidebar, properties panel, and console
  • Canvas (center) — The React Flow-powered visual editor where you design pipelines by placing and connecting nodes. Supports zoom, pan, minimap, and multi-select.
  • Sidebar (left) — Three tabs: Explorer (pipeline list), Nodes (component palette), and Resources (connections and variables).
  • Properties Panel (right) — Configuration panel that appears when you select a node. Shows all settings specific to that node type.
  • Console (bottom) — Execution log showing real-time progress, row counts, errors, and timing information.
  • Tab Bar (top) — Open multiple pipelines in tabs. Each tab maintains its own independent state.
  • Toolbar — Run, Abort, Save, Undo, and other action buttons.

3.2 Explorer

Explorer panel showing MY MAESTROS and MY PIPELINES with item counts

The Explorer tab in the left sidebar organizes your project items into three collapsible sections:

  • MY MAESTROS — Pipeline orchestration workflows
  • MY PIPELINES — Data transformation pipelines

Each section shows a count badge (e.g., "3") and a search field for filtering. Right-click any item for context menu options: Rename, Duplicate, Delete. Double-click to open in a new tab.

3.3 Nodes Palette

Complete node catalog with INGEST (22 nodes), TRANSFORM (5 nodes), and DELIVER (29 nodes)

The Nodes tab in the left sidebar presents the complete catalog of available components, organized into three color-coded categories:

INGEST (22 nodes)

Data sources: files (CSV, Excel, XML, Magic File), databases (PostgreSQL, MySQL, Oracle, MSSQL, MongoDB, Snowflake, DB2, ODBC), cloud storage (S3, Google Drive, Azure Blob), remote (FTP/SFTP, REST Source), local (File List)

TRANSFORM (5 nodes)

SQL Transform, Python Transform, Filter, Mapper, Set Variable

DELIVER (29 nodes)

File targets, database targets, cloud storage targets, FTP targets, REST Target, Email Target, Console, Folder/File Management, control flow (Sleep, RunAfter, LoopWhile)

3.4 Adding Nodes

There are two ways to add nodes to the canvas:

  • Drag and drop — Click and drag a node from the Nodes palette onto the canvas. Release the mouse to place the node at the desired position.
  • AI generation — Use the AI Assistant to describe your pipeline in natural language. The AI will generate nodes and connections automatically.

3.5 Connecting Nodes

Nodes communicate through directed edges that define the data flow. To create a connection:

  1. Hover over the output port (small circle on the right side of a node). The port highlights when hoverable.
  2. Click and drag from the output port toward the target node.
  3. Release on the input port (small circle on the left side of the target node).

A colored edge appears connecting the two nodes, indicating data will flow from source to target during execution. You can delete an edge by clicking on it and pressing Delete.

Tip Nodes can have multiple input and output connections. SQL Transform, for instance, can accept multiple inputs and reference each by name in the query.

3.6 Configuring Nodes

Click any node on the canvas to open its Properties panel on the right side. Each node type has specific configuration fields.

Properties panel for a Python Transform node showing code editor and configuration options

Common properties shared by most nodes include:

  • Name — Display label on the canvas. Also used as the table name in SQL transforms.
  • Description — Optional text for documentation purposes.
  • Active — Toggle to enable/disable the node during execution.

3.7 Multi-Tab Support

Odara supports opening multiple pipelines and maestros simultaneously in separate tabs. Each tab maintains its own independent state including:

  • Canvas zoom level and pan position
  • Selected node and properties panel
  • Console log entries
  • Execution state (running, completed, failed)

Tabs are displayed in the tab bar at the top of the editor. Click a tab to switch to it. Close a tab by clicking the X icon. Unsaved changes are indicated with a dot on the tab.

3.9 Preview Output

Execution preview showing output data in a tabular format with row count

During or after execution, you can preview the output of any node by clicking the preview button. The preview panel shows:

  • A data table with column headers and typed values
  • Row count indicator (e.g., "Showing 20 of 1,453 rows")
  • Column data types
  • Scrollable table for wide datasets

Preview data is sampled from the node's output during execution and stored for later inspection. This allows you to verify transformations without running the entire pipeline again.

3.10 Execution

To execute a pipeline, click the Run button in the toolbar. Execution uses Server-Sent Events (SSE) for real-time progress streaming. As each node executes:

  • The node border turns blue (currently executing)
  • On success, the node turns green with a checkmark
  • On failure, the node turns red with an error icon
  • Skipped nodes (inactive or unreachable) remain gray

The console at the bottom shows detailed step-by-step logs with timestamps, row counts, and duration for each node. If a node fails, the error message and stack trace appear in the console.

You can abort a running pipeline at any time by clicking the Abort button. This sends a cancellation signal that gracefully stops execution after the current node completes.

3.11 Console Log

The console panel at the bottom of the editor provides a chronological log of execution events. Each entry includes:

  • Timestamp — When the event occurred
  • Severity — Info (blue), Warning (yellow), or Error (red)
  • Source — Either "Execution" (pipeline-level) or the specific node name
  • Message — Description of what happened, including row counts and durations

Controls: Filter by All or Errors only; Auto-scroll toggle to follow new entries; Clear to reset the log.

3.12 Pipeline Screenshots

Every time you save a pipeline, Odara automatically captures a screenshot of the current canvas state. These screenshots are stored in {project_path}/data/screenshots/pipeline_{id}.png and are used for:

  • Auto-generated project documentation
  • Execution history visual reference
  • Quick preview in the Explorer

3.13 Data Quality Tests

Data Quality Tests panel showing the empty state with an "Add Test" button
Dialog for adding a new data quality test with type, column, condition, severity, and threshold

Data quality tests validate your data at specific points in the pipeline. You can add tests to any node by opening the "Data Quality" tab in the properties panel.

Available test types:

TypeDescriptionExample
Not NullEnsures a column has no null valuesColumn "email" must not be null
UniqueEnsures all values in a column are distinctColumn "customer_id" must be unique
RangeValidates values fall within a min/max rangeColumn "age" between 0 and 150

Each test has a severity level: Error (stops execution), Warning (logs but continues), or Info (informational only). You can also set a threshold percentage to allow a small number of failures (e.g., "allow up to 1% null values").

4.1.4 Magic File SOURCE

Magic File auto-detects the format and reads the data without requiring format-specific configuration

Magic File is an intelligent file reader that automatically detects the file format based on the extension and content. It supports CSV, JSON, NDJSON (newline-delimited JSON), Parquet, Excel, and XML. Simply provide the file path and Magic File handles the rest.

This is especially useful in pipelines that process files from external sources where the format may vary, or when you want to keep configuration minimal.

4.2.5 MongoDB Source SOURCE

MongoDB Source with collection name, filter JSON, and aggregation pipeline options

Connects to MongoDB and reads documents from a collection. Unlike SQL databases, MongoDB Source offers two modes of querying:

Documents are automatically flattened into a tabular format. Nested objects become dot-notation columns (e.g., address.city).

4.6 Transforms

4.6.1 SQL Transform TRANSFORM

SQL Transform with Monaco code editor, syntax highlighting, and multi-input table references

SQL Transform is the primary transformation node in Odara. It executes SQL queries using the Apache DataFusion engine, which provides a full SQL dialect operating on Apache Arrow columnar data.

The input data from upstream nodes is registered as tables. By default, a single input is available as the input table. When multiple nodes connect to a SQL Transform, each is accessible by its node name.

-- Single input: reference as 'input'
SELECT category,
       COUNT(*) AS product_count,
       AVG(price) AS avg_price,
       SUM(stock) AS total_stock
FROM input
GROUP BY category
ORDER BY product_count DESC
-- Multiple inputs: reference by node name
SELECT o.order_id, o.amount, c.name, c.email
FROM orders_csv o
JOIN customers_db c ON o.customer_id = c.id
WHERE o.amount > 100

Supported SQL features:

  • SELECT, FROM, WHERE, GROUP BY, ORDER BY, HAVING, LIMIT
  • JOINs: INNER, LEFT, RIGHT, FULL OUTER, CROSS
  • Subqueries (correlated and uncorrelated)
  • Common Table Expressions (WITH ... AS)
  • UNION, UNION ALL, INTERSECT, EXCEPT
  • Window functions: ROW_NUMBER, RANK, DENSE_RANK, LAG, LEAD, SUM OVER, etc.
  • CASE expressions
  • String, math, date/time, and aggregate functions
  • CAST and type conversion

Additional options:

  • Pre-SQL — SQL executed before the main query (e.g., creating temporary views)
  • Post-SQL — SQL executed after the main query
  • Variable interpolation — Use @variable_name to inject context variable values into SQL

4.6.2 Python Transform TRANSFORM

Python Transform code editor with syntax highlighting and pandas DataFrame support

Python Transform executes custom Python code with full access to the pandas ecosystem. The input data arrives as a pandas DataFrame named df. Your code must return a DataFrame.

# Input data is available as 'df' (pandas DataFrame)
# You can use any pandas operations

# Add calculated columns
df['profit_margin'] = (df['price'] - df['cost']) / df['price'] * 100
df['amount_with_tax'] = df['amount'] * 1.2

# Filter and transform
df = df[df['profit_margin'] > 0]
df['category'] = df['category'].str.upper()

# Always return the DataFrame
return df

Python Transform runs in a subprocess, so it does not affect the main Rust process. Data is exchanged using PyArrow for efficient serialization. You can import any Python library installed in your environment (numpy, scikit-learn, etc.).

4.6.3 Filter TRANSFORM

Filter node with a SQL WHERE condition to filter rows

Filter applies a SQL WHERE condition to the incoming data, keeping only rows that match the condition. It is a lightweight alternative to SQL Transform when you only need to filter rows without transforming columns.

-- Example filter conditions
status = 'completed' AND CAST(amount AS DOUBLE) > 100

-- Date filtering
created_at >= '2024-01-01' AND created_at < '2025-01-01'

-- Pattern matching
email LIKE '%@company.com'

4.6.4 Mapper TRANSFORM

The Mapper is Odara's equivalent of Talend's tMap component. It provides a visual interface for complex data routing with:

  • N inputs — Accept data from multiple upstream nodes
  • M outputs — Route data to multiple downstream nodes
  • Joins — Join inputs on key columns
  • Expressions — SQL expressions for column transformations
  • Column mapping — Drag columns from inputs to outputs
  • Filtering — Apply conditions to each output

Open the Mapper editor by double-clicking the Mapper node. The visual editor shows input tables on the left, output tables on the right, and join configuration in the center.

Mapper workflow:

  1. Connect multiple source nodes to the Mapper's input ports
  2. Double-click the Mapper to open the visual editor
  3. Configure joins between input tables by selecting key columns
  4. Drag columns from input tables to output tables to create mappings
  5. Add SQL expressions for computed columns (e.g., CONCAT(first_name, ' ', last_name))
  6. Add filter conditions on each output table to route different subsets of data
  7. Connect each Mapper output port to a downstream target node
Tip The Mapper is ideal for scenarios where you need to split incoming data into multiple streams based on conditions, similar to a conditional router. Each output can have its own filter, so a single Mapper can replace multiple Filter nodes.

4.6.5 Set Variable TRANSFORM

Set Variable captures a value from the data stream and stores it as a context variable. This variable can then be used in downstream nodes via the @variable_name syntax.

Common use cases include:

  • Incremental loads — Query SELECT MAX(updated_at) FROM target_table, store as @last_sync, then use WHERE updated_at > '@last_sync' in the source query
  • Row counting — Store a count from one branch and use it in email notifications or conditional logic
  • Dynamic filenames — Capture a value (e.g., report date) and use it in output file paths: report_@report_date.csv
  • Cross-branch communication — Pass computed values between parallel pipeline branches using shared context variables

Set Variable evaluates an expression against the first row of the incoming data. The expression can be a column name (to capture that column's value) or a SQL expression.

4.7 Control Flow

4.7.1 Sleep CONTROL

Pauses pipeline execution for a specified number of seconds. Useful for rate-limiting API calls, waiting for external processes, or introducing delays in testing.

The delay value supports variable interpolation: set it to @delay_seconds for dynamic delays based on context variables.

4.7.2 RunAfter CONTROL

RunAfter is a synchronization point that waits for all upstream nodes to complete before allowing downstream execution to proceed. It does not transform data; it only controls execution order.

This is essential when you have parallel branches that must all finish before a subsequent step begins (e.g., loading data from multiple sources before running a consolidation query).

4.7.3 LoopWhile CONTROL

LoopWhile repeats execution of its downstream nodes as long as a SQL condition evaluates to true. This enables iterative processing patterns such as:

  • Paginating through an API until no more data is returned
  • Retrying a failed operation with a counter
  • Processing data in batches until a threshold is met

The loop condition is a SQL expression evaluated against context variables. For example:

-- Loop while there are more pages to fetch
@has_more_pages = 'true'

-- Loop with a counter (max 10 iterations)
CAST(@iteration_count AS INT) < 10

Inside the loop body, use a Set Variable node to update the loop condition variable. If the condition is never set to false, add a maximum iteration limit to prevent infinite loops.

5. Resources

5.1 Overview

Resources panel organized into Data Stores, File Storage, Schemas, and Runtime sections

The Resources tab in the left sidebar provides centralized management of reusable configurations. Resources are shared across all pipelines in a project, eliminating the need to re-enter connection details for every node.

Resources are organized into four categories:

  • Data Stores — Database connections (PostgreSQL, MySQL, Oracle, SQL Server, MongoDB, Snowflake, DB2, ODBC)
  • File Storage — Cloud and remote storage connections (S3, Google Drive, Azure Blob, FTP/SFTP)
  • Schemas — Reusable file format definitions
  • Runtime — Context variables and environments

5.2 Data Stores

Database connection dialog with type, host, port, database, credentials, and Test Connection button

Data Stores define reusable database connections. When configuring a database source or target node, you can select a Data Store from the dropdown instead of entering connection details manually.

Creating a Data Store:

  1. Click the + button next to "Data Stores" in the Resources panel
  2. Select the database type (PostgreSQL, MySQL, Oracle, SQL Server, MongoDB, Snowflake, DB2, ODBC)
  3. Fill in connection details: host, port, database name, schema (if applicable), username, and password
  4. Click Test Connection to verify connectivity
  5. Click Save to store the connection

Schema Introspection: Once a Data Store is saved and connected, you can introspect its schema. This shows all tables, views, and their columns with data types. Click any table to see its column list, which can be referenced when writing SQL queries.

Security All credentials (passwords, API keys, tokens) are encrypted at rest using AES-GCM encryption. They are never stored in plain text in the SQLite database.

5.3 File Storage

File Storage resources define connections to cloud storage and remote file servers. These include:

  • S3 — Region, access key, secret key, optional endpoint URL for S3-compatible services
  • Google Drive — OAuth2 client ID, client secret, refresh token
  • Azure Blob Storage — Account name, SAS token or connection string
  • FTP/SFTP/FTPS — Protocol, host, port, username, password

5.4 Schemas

Schema resources define reusable file format specifications. Instead of configuring column names, types, and delimiters on every file node, define a schema once and reference it across multiple pipelines.

A schema definition includes:

  • Column names and data types (String, Integer, Float, Boolean, Date, Timestamp)
  • Format-specific settings (CSV delimiter, XML row element, JSON data path)
  • Null handling and default values

5.5 Variables

Context variables are named values stored in the Runtime section of Resources. They can be referenced throughout your pipelines using the @variable_name syntax.

TypeDescriptionExample

5.6 Environments

Environments allow you to define different variable values for different stages of your workflow (e.g., Dev, QA, Prod). Each environment can override any context variable.

For example, you might have:

  • Dev: @db_host = "localhost", @batch_size = 100
  • QA: @db_host = "qa-db.internal", @batch_size = 1000
  • Prod: @db_host = "prod-db.internal", @batch_size = 10000

Switch the active environment using the environment selector. All @variable references automatically resolve to the values defined for the active environment.

5.7 Using Variables

Variables can be used anywhere in Odara that accepts text input. The @variable_name syntax is replaced with the variable's value at execution time.

Where variables work:

  • SQL queries: WHERE created_at >= '@cutoff_date'
  • File paths: /data/@environment/output_@date.csv
  • Connection strings: @db_host:@db_port
  • REST URLs: https://api.example.com/v1/@api_version/users
  • Email fields: Subject: Daily Report for @date
  • Sleep duration: @delay_seconds

6. Maestro (Orchestration)

6.1 What is Maestro

Maestro is Odara's pipeline orchestration layer. While a pipeline defines a single data flow (source to target), Maestro coordinates the execution of multiple pipelines with dependencies, parallel execution, conditional branching, and retry policies.

Think of Maestro as a "pipeline of pipelines." It answers questions like: "Run the customer ETL first, then the orders ETL, and if both succeed, run the report generation in parallel with the data quality check."

6.2 Canvas

The Maestro canvas uses the same React Flow editor as the pipeline canvas, but with orchestration-specific step types instead of data nodes. Steps are connected with edges that define the execution order and dependency graph.

Create a new Maestro from the Explorer panel under "MY MAESTROS". The Maestro canvas opens in a new tab, independent of any pipeline tabs.

6.3 Step Types

Pipeline Call

Execute a child pipeline by reference. Select any pipeline in the project. The step completes when the pipeline execution finishes. If the pipeline fails, the step fails (unless retry is configured).

Parallel Group

Execute multiple steps simultaneously. Configure max concurrency to limit how many steps run at the same time. Useful for independent pipelines that can run in parallel.

Series Group

Execute steps sequentially, one after another. Each step waits for the previous one to complete before starting. If any step fails, subsequent steps are skipped.

Conditional

Branch execution based on a SQL condition. Define a true branch and a false branch. The condition is evaluated against context variables and previous step results.

6.4 Retry Policies

Each Maestro step can have an individual retry policy:

SettingDescriptionDefault
Max AttemptsMaximum number of execution attempts1 (no retry)
Initial DelayWait time before first retry (seconds)5
Backoff MultiplierMultiply delay by this factor for each subsequent retry2.0 (exponential)

For example, with max_attempts=3, initial_delay=5s, and backoff=2.0: the first retry happens after 5 seconds, the second after 10 seconds.

6.5 Built-in Variables

Maestro provides additional built-in variables scoped to the orchestration execution:

VariableDescription
${pipeline.execution_id}Execution ID of the most recently completed pipeline step
${step.previous_result}Status of the previous step (success/failed/skipped)
${maestro.name}Name of the Maestro workflow
${step.name}Name of the current step
${step.index}Zero-based index of the current step in execution order

6.6 Running

Execute a Maestro by clicking the Run button in the Maestro canvas toolbar. Execution progress is streamed via SSE, showing each step's status in real-time on the canvas. You can abort a running Maestro at any time, which sends cancellation signals to all active pipeline executions.

Monitor Maestro executions in the Monitor page (see Section 9).

8. Schedule

8.1 Overview

Schedule management page with list of scheduled jobs, cron expressions, and status

The Schedule page allows you to set up automated, recurring executions of your pipelines and maestros. Schedules are defined using cron expressions and can be managed through a visual builder or by entering a custom expression directly.

8.2 Cron Builder

The cron builder provides two modes:

Visual mode offers preset patterns:

  • Every Minute — * * * * *
  • Hourly — At a specific minute (e.g., 30 * * * * for half past every hour)
  • Daily — At a specific time (e.g., 0 6 * * * for 6:00 AM daily)
  • Weekly — On selected days at a specific time (e.g., 0 9 * * 1-5 for weekdays at 9 AM)
  • Monthly — On a specific day at a specific time (e.g., 0 2 1 * * for 2 AM on the 1st)

Custom mode accepts a full 6-field cron expression:

second minute hour day_of_month month day_of_week
  0      30     6       *         *       *        # 6:30 AM daily
  0       0     */2     *         *       *        # Every 2 hours
  0      15     10      1,15      *       *        # 10:15 AM on 1st and 15th

8.3 Timezone

Each schedule can be assigned a timezone. The default is UTC. Available timezones include:

  • UTC, US/Eastern, US/Central, US/Mountain, US/Pacific
  • Europe/London, Europe/Paris, Europe/Berlin
  • Asia/Tokyo, Asia/Shanghai, Australia/Sydney

8.4 Associate

A schedule must be associated with either a pipeline or a maestro. Select the target from a dropdown that lists all available pipelines and maestros in the current project. When the schedule triggers, it creates a new execution of the associated item.

8.5 History

Each schedule maintains a history of triggered executions showing start time, duration, and status (success/failed/aborted). This helps you monitor the health of automated workflows and identify recurring failures.

15.3 DataFusion SQL

Odara uses Apache DataFusion as its SQL engine. DataFusion provides a full ANSI SQL dialect operating on Apache Arrow columnar data in memory.

Supported SQL statements:

  • SELECT, FROM, WHERE, GROUP BY, ORDER BY, HAVING, LIMIT
  • JOIN (INNER, LEFT, RIGHT, FULL OUTER, CROSS)
  • UNION, UNION ALL, INTERSECT, EXCEPT
  • Subqueries (scalar, correlated, EXISTS)
  • Common Table Expressions (WITH ... AS)
  • Window functions (ROW_NUMBER, RANK, DENSE_RANK, LAG, LEAD, SUM OVER, AVG OVER)
  • CASE expressions
  • CAST and type conversion

15.4 Variable Interpolation

Odara supports variable interpolation in most text fields using the @variable_name syntax. At execution time, @variable_name is replaced with the variable's current value.

-- In file paths

-- In REST URLs
https://api.example.com/@api_version/users?page=@page_number

15.8 Troubleshooting

Odara does not start

Check that port 3002 is not already in use: lsof -i :3002 (Linux/macOS) or netstat -an | findstr 3002 (Windows). If another process is using the port, stop it or start Odara on a different port with odara --port 8080.

Browser shows blank page or connection refused

Ensure Odara is running — check the terminal for errors. Open http://localhost:3002 (not 3000 or 5175). If the browser shows "connection refused", the server may have crashed on startup — check the terminal output for error messages.

Database connection fails

Test the connection using the "Test Connection" button in the metadata dialog. Verify the database service is running and accessible. Check firewall rules if connecting to a remote database. For Docker services, ensure the container is running with docker ps.

Python Transform fails

Ensure Python 3.9+ is installed and available in the PATH. Install the required package with: pip install pyarrow. On Windows, make sure to disable the Microsoft Store Python alias (see Installation section).

SQLite database is locked

SQLite uses WAL mode for concurrent access, but only one writer is allowed at a time. If you see "database is locked" errors, check for multiple API server instances running on the same database file. Stop duplicate processes.

Pipeline execution hangs

Use the Abort button to cancel the execution. Check the console for the last logged step. Common causes: database queries without timeouts, REST API calls to unresponsive servers, or infinite LoopWhile conditions.

Where are my files?

  • SQLite database: {project_path}/data/odara.db
  • Pipeline screenshots: {project_path}/data/screenshots/
  • API logs: Standard output of the API server process
  • Frontend logs: Browser developer console (F12)

Oracle connector build fails

The Oracle connector uses the sibyl crate which requires Oracle Instant Client libraries. Ensure you have installed the Oracle Instant Client (Basic + SDK) for your platform and set the ORACLE_HOME and LD_LIBRARY_PATH (Linux) or DYLD_LIBRARY_PATH (macOS) environment variables.

export ORACLE_HOME=/opt/oracle/instantclient
export LD_LIBRARY_PATH=$ORACLE_HOME:$LD_LIBRARY_PATH

SFTP connection refuses

Verify the target server allows password authentication (some servers require key-based auth only). Check that the port is correct (default 22 for SFTP). If using Docker, the SFTP container maps to port 2222 by default: docker-compose -f docker-compose.services.yml up -d sftp.

REST Source returns empty data

Check the Data Path setting. If the API response wraps data in an object (e.g., {"data": [...]}), set Data Path to $.data. Enable the browser network tab to inspect the raw response. Also verify that the authentication credentials are correct and the endpoint returns data for the given parameters.

Azure features not available

Azure Blob Storage nodes are only available in the Azure-enabled build variant. Download the Azure build from the Releases page.

Performance tuning for large datasets

For pipelines processing millions of rows:

  • Use Parquet format instead of CSV for intermediate files (faster read/write, columnar compression)
  • Increase database target batch size (e.g., 10000 rows per batch) for bulk inserts
  • Use SQL Transform with DataFusion instead of Python Transform for aggregations (native Rust speed)
  • Consider splitting large pipelines into multiple Maestro steps to manage memory usage
  • Enable WAL mode on SQLite (enabled by default) for concurrent read performance

Resetting the project database

If your project database becomes corrupted or you want to start fresh, stop the API server and delete the SQLite files:

rm {project_path}/data/odara.db
rm {project_path}/data/odara.db-wal
rm {project_path}/data/odara.db-shm

The database will be recreated automatically when the API server starts. Note that this deletes all pipelines, metadata, execution history, and variables in the project.
